# 시설 운영 시스템 민원 통합 설계안

## 목적

현재 시스템은 시설, 재고, 작업지시, 보고서, 권한관리를 중심으로 운영된다.
민원처리시스템을 여기에 통합할 때는 `민원 접수`와 `현장 실행`을 같은 개념으로 보지 않는 것이 핵심이다.

권장 방향은 다음과 같다.

- `민원`은 접수, 분류, 회신, 만족도까지 관리하는 고객 접점 레이어
- `작업지시`는 실제 현장 조치와 자재 투입, 담당자 배정, 완료 이력을 관리하는 실행 레이어
- 두 레이어는 연결되지만 서로 대체하지 않는다

## 현재 구조에서 가능한 이유

기존 시스템에는 이미 민원을 흡수할 기반이 있다.

- `work_orders.category`에 `민원` 분류가 존재한다
- `requester_name`, `priority`, `status`, `assignee_user_id`, `due_date`가 있다
- 첨부파일과 작업 업데이트 이력이 있다

즉, 지금도 민원성 작업을 등록할 수는 있다.
다만 접수번호, 접수채널, 동호 정보, 민원인 연락처, 회신 이력, 만족도, 재오픈 이력은 없다.
그래서 `작업지시 안에 민원을 억지로 넣는 방식`보다 `민원 모듈을 추가하고 작업지시와 연결`하는 편이 맞다.

## 통합 원칙

### 1. 민원과 작업지시를 분리한다

- 민원은 고객 관점의 요청 단위다
- 작업지시는 운영 관점의 실행 단위다
- 민원 1건이 작업지시 0건, 1건, 여러 건으로 이어질 수 있다

### 2. 민원 종결과 작업 완료를 분리한다

- 작업 완료: 현장 조치가 끝난 상태
- 민원 종결: 조치 결과 회신과 확인까지 끝난 상태

### 3. 민원 이력은 삭제보다 상태 전환 중심으로 관리한다

- 접수
- 분류
- 배정
- 처리중
- 처리완료
- 회신완료
- 종결
- 재오픈

### 4. 운영 보고서는 민원 기준과 작업 기준을 둘 다 본다

- 민원 보고서: 접수량, 지연, 재접수, 만족도
- 작업 보고서: 처리 시간, 담당자, 자재 사용, 시설 유형

## 권장 데이터 모델

### 신규 테이블

#### `complaints`

민원 마스터 테이블.

- `id`
- `complaint_code`
- `channel`
  예: 전화, 방문, 모바일, 카카오톡, 이메일
- `category_primary`
  예: 소음, 누수, 전기, 설비, 청소, 주차, 기타
- `category_secondary`
- `facility_id`
- `unit_label`
  예: 101동 1203호
- `location_detail`
- `requester_name`
- `requester_phone`
- `requester_email`
- `title`
- `description`
- `priority`
- `status`
- `received_at`
- `response_due_at`
- `resolved_at`
- `closed_at`
- `assignee_user_id`
- `created_by`
- `updated_by`
- `created_at`
- `updated_at`

#### `complaint_updates`

민원 상태 변경, 회신, 내부 메모, 대외 안내 기록.

- `id`
- `complaint_id`
- `update_type`
  예: 메모, 상태변경, 회신, 재오픈, 취소
- `status_from`
- `status_to`
- `message`
- `is_public_note`
- `created_by`
- `created_at`

#### `complaint_feedback`

민원 처리 후 만족도.

- `id`
- `complaint_id`
- `score`
  1~5점
- `comment`
- `submitted_at`

### 기존 테이블 확장

#### `work_orders`

기존 작업지시에 민원 연결 필드를 추가한다.

- `complaint_id INTEGER NULL`

이렇게 하면 한 민원에서 여러 작업지시를 만들 수 있다.
각 작업지시는 최대 1개의 민원에 소속되게 두는 편이 운영상 단순하다.

#### `attachments`

첨부가 민원에도 붙을 수 있게 대상 타입을 일반화하는 편이 좋다.

권장 확장:

- `entity_type`
  예: `work_order`, `complaint`
- `entity_id`

현재 구조를 크게 깨고 싶지 않다면 1차에서는 `complaint_attachments` 별도 테이블을 둬도 된다.

## 권장 상태 모델

### 민원 상태

- `접수`
- `분류완료`
- `배정완료`
- `처리중`
- `처리완료`
- `회신완료`
- `종결`
- `보류`
- `취소`
- `재오픈`

### 작업지시 상태

기존 상태를 유지한다.

- `접수`
- `진행중`
- `대기`
- `보류`
- `완료`
- `종결`

민원과 작업지시 상태를 1:1로 강제 연동하지 않는다.
대신 아래처럼 연계 규칙을 둔다.

- 민원 `배정완료` 시 작업지시 자동 생성 가능
- 연결된 작업지시가 모두 `완료`면 민원 상태를 `처리완료` 후보로 표시
- 담당자가 회신을 입력하면 민원 상태를 `회신완료`
- 최종 확인 후 `종결`

## 권한 모델 확장

현재 역할은 `admin`, `manager`, `technician`, `viewer`로 충분하다.
1차에서는 새 역할을 만들지 않고 민원 권한만 추가하는 편이 낫다.

권장 권한:

- `complaints:view`
- `complaints:create`
- `complaints:edit`
- `complaints:assign`
- `complaints:update`
- `complaints:close`
- `complaints:feedback:view`

권장 매핑:

- `admin`: 전체 권한
- `manager`: 민원 전체 관리, 배정, 회신, 종결
- `technician`: 본인 배정 민원 조회, 작업진행 입력, 내부 메모 작성
- `viewer`: 민원 조회만

## 화면 설계

### 1. 민원 대시보드

- 오늘 접수 건수
- 미배정 민원
- SLA 초과 민원
- 재오픈 민원
- 만족도 평균
- 시설별 반복 민원 TOP

### 2. 민원 접수함

- 검색: 민원번호, 제목, 동호수, 민원인, 전화번호
- 필터: 상태, 채널, 유형, 담당자, 접수일
- 빠른 작업: 배정, 우선도 변경, 작업지시 생성

### 3. 민원 상세 화면

- 민원 기본 정보
- 회신 이력
- 작업지시 연결 목록
- 사진/첨부
- 만족도 결과

### 4. 작업지시 상세 화면

- 연결된 민원 번호 표시
- 민원 요약 표시
- 작업 완료 시 민원 회신 초안 작성 버튼

### 5. 보고서

- 기간별 민원 접수/종결 수
- 평균 1차 응답 시간
- 평균 처리 완료 시간
- 유형별 민원 분포
- 시설/동호 반복 민원
- 담당자별 처리 편차

## 추천 업무 흐름

### 표준 흐름

1. 민원 접수
2. 유형 분류 및 우선도 판단
3. 담당자 또는 파트 배정
4. 필요 시 작업지시 생성
5. 현장 조치
6. 조치 결과 회신
7. 만족도 기록
8. 민원 종결

### 단순 문의 흐름

현장 출동이 필요 없는 민원은 작업지시 없이 민원만 종결할 수 있어야 한다.

예:

- 사용 문의
- 안내 요청
- 단순 정책 문의

## 데이터 이관 전략

현재 `work_orders` 중 `category='민원'` 인 데이터는 초기 민원 데이터로 부분 이관할 수 있다.

권장 매핑:

- `work_code` -> 임시 민원 참조번호
- `requester_name` -> `complaints.requester_name`
- `title` -> `complaints.title`
- `description` -> `complaints.description`
- `priority` -> `complaints.priority`
- `status` -> 상태 규칙에 따라 매핑
- `facility_id` -> `complaints.facility_id`
- `assignee_user_id` -> `complaints.assignee_user_id`

이관 후에는 기존 작업지시에 `complaint_id`를 채워 연결한다.

## 구현 순서

### 1차

- `complaints`
- `complaint_updates`
- 민원 목록/상세/등록/수정
- 작업지시와 민원 연결
- 민원 대시보드 기본 카드

### 2차

- `complaint_feedback`
- 회신 템플릿
- 반복 민원 탐지
- SLA 초과 경고

### 3차

- 외부 접수 폼
- 입주민 공개용 진행 조회
- SMS/카카오톡 회신 연동
- 만족도 자동 요청

## 구현 시 주의점

- 민원 번호와 작업 번호를 혼동하지 않게 별도 코드 체계를 둔다
- 민원 삭제는 최소화하고 비공개/취소/오입력 상태로 처리한다
- 첨부파일은 민원과 작업지시에서 중복 업로드하지 않도록 참조를 정리한다
- 보고서는 민원 관점과 작업 관점을 섞지 말고 분리 집계한다

## 결론

민원처리시스템은 시설 운영 시스템에 충분히 녹여 넣을 수 있다.
다만 `민원 = 작업지시`로 합치는 방식보다 `민원 접수 모듈 + 작업지시 실행 모듈` 구조가 더 정확하고 확장성이 높다.

현재 코드베이스에서는 이 방식이 가장 안전하다.
