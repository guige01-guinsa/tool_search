# 시설 운영 시스템 전면 재설계 계획

## 목적

기존 `tool_search`는 공구 이미지 검색과 단일 운영자 중심 흐름에 최적화되어 있었다.
새 시스템은 다음 5개 축을 하나의 운영 플랫폼으로 통합한다.

- 시설 관리
- 재고 관리
- 작업지시 관리
- 운영 보고서
- 권한 관리

목표 사용자는 9명 규모의 현장 운영팀이며, 공용 웹 환경에서 동시 사용 가능한 구조를 전제로 한다.

## 기존 구조의 한계

- `main.py` 중심의 단일 파일 구조라 기능 확장 시 결합도가 높다.
- 로그인과 권한 분리가 없어 9명 공유 환경에서 수정 책임을 추적하기 어렵다.
- 공구 데이터는 존재하지만 시설, 작업지시, 보고서 체계가 분리되어 있다.
- 재고가 단순 마스터 수준에 머물러 수불 이력이 운영 자산으로 누적되지 않는다.

## 새 구조

### 앱 엔트리포인트

- `ops_main.py`

### 모듈

- `ops/db.py`
  운영 시스템 스키마 초기화, SQLite 설정, 레거시 데이터 이관
- `ops/auth.py`
  PBKDF2 기반 비밀번호 해시, 세션 발급, 역할/권한 정의
- `ops/ui.py`
  공통 레이아웃, 상태 배지, 카드, 선택 옵션 렌더링

### 데이터 저장

- 신규 DB: `operations.db`
- 첨부 파일: 기존 `uploads/` 폴더 재사용
- 레거시 DB 이관: 필요할 때만 `LEGACY_DB_PATH` 환경변수로 지정
- 배포 환경에서는 `OPS_DB_PATH`, `OPS_UPLOAD_DIR`, `OPS_ADMIN_PASSWORD`, `OPS_COOKIE_SECURE` 환경변수로 운영값을 주입

## 레거시 데이터 이관 전략

### 재고

레거시 백업 DB를 지정하면 `tools` 데이터를 `inventory_items`로 이관한다.

- `name` -> 품목명
- `purpose` -> 규격/사양
- `qty` -> 수량
- `location` -> 보관 위치
- `status` -> 상태
- `purchase_amount` -> 구매금액
- `cat_l/m/s` -> 분류 문자열

### 시설

레거시 `location` 고유값을 추출해 `facilities` 기본 데이터로 생성한다.

### 첨부

`tool_images`의 파일명을 `attachments`로 연결한다.

### 이력

`tool_events`는 `inventory_transactions`로 가져오되, 수량 증감이 불명확한 건 `quantity_delta=0`으로 저장한다.

## 구현된 1차 범위

- 로그인 / 로그아웃
- 역할 기반 화면 접근 제어
- 대시보드
- 시설 CRUD
- 재고 CRUD
- 재고 수불 이력
- 작업지시 CRUD
- 작업 업데이트 이력
- 기간 기반 운영 보고서
- 사용자/권한 관리
- 레거시 공구 데이터 자동 이관

## 운영 전환 순서

1. 새 앱을 `ops_main.py`로 배포한다.
2. 관리자 계정으로 로그인 후 실제 사용자 9명을 등록한다.
3. 초기 관리자 비밀번호를 즉시 변경한다.
4. 시설 분류와 상태값을 현장 기준으로 정리한다.
5. 레거시에서 넘어온 재고 중 불필요한 품목과 위치명을 정제한다.
6. 이후 작업지시를 실사용하면서 보고서 포맷을 현장 문서 양식에 맞춘다.
